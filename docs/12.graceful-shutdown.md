## Overview

ecewo handles graceful shutdown automatically. When the process receives `SIGINT` (Ctrl+C) or `SIGTERM` (e.g. `kill`), the server:

1. Stops accepting new connections
2. Closes idle keep-alive connections immediately
3. Waits for in-progress requests to finish (up to the configured `shutdown_timeout_ms`, default 15 seconds)
4. Force-closes any connections still open after the timeout

You do not need to set up signal handlers yourself.

## Shutdown Sequence

```
Signal received (SIGINT / SIGTERM), or server_shutdown() called
  │
  ├─ server_is_running() returns false
  ├─ Stop accepting new connections
  ├─ Wait for pending async work (spawn(), spawn_http(), or external functions coming from plugins) to complete
  ├─ Wait for in-progress requests to finish
  ├─ Force-close remaining connections
  │
  ├─ server_atexit callback fires
  │    All requests are done. Event loop is still alive.
  │    Safe to close DB connections, pools, files, and free the resources.
  │
  ├─ Event loop teardown
  └─ Event loop freed
       │
server_run() returns
```

## Resource Cleanup

Register a callback with `server_atexit()` to release all resources your application uses. The callback fires after every in-flight request has finished and while the event loop is still alive, making it the single correct place for all cleanup; whether your resources are plain C objects, database connection pools, or plugins that rely on libuv internally.

```c
#include "ecewo.h"
#include "ecewo-session.h"
#include "ecewo-cors.h"
#include "db.h"

void on_shutdown(void) {
  db_cleanup();
  cors_cleanup();
  session_cleanup();
}

int main(void) {
  App *app = ecewo();
  if (!app) {
    fprintf(stderr, "Failed to initialize server\n");
    return -1;
  }

  session_init();
  cors_init(&cors);
  db_init();

  server_atexit(app, on_shutdown);

  get(app, "/users", get_users);
  post(app, "/users", create_user);

  if (server_listen(app, 3000) != 0) {
    fprintf(stderr, "Failed to start server\n");
    return -1;
  }

  server_run(app);

  return 0;
}
```

> [!WARNING]
>
> Do not place cleanup code after `server_run()`. By the time `server_run()` returns, the event loop has already been freed.

## Intentional Shutdown

To trigger shutdown from inside a route handler, call `server_shutdown()` via `req->app`:

```c
void shutdown_handler(Req *req, Res *res) {
	send_text(res, OK, "Server is shutting down");
	server_shutdown(req->app);
}
```

This is safe to call from within a handler. The server finishes the current request normally, then proceeds with the shutdown sequence.

> [!NOTE]
>
> **NOTE FOR PLUGIN DEVELOPMENT:**
>
> Processes that check `server_is_running()` do not need `server_atexit` to receive the stop signal. `server_is_running()` returns `false` as soon as shutdown begins, before any cleanup runs.
>
> Additionally, increment_async_work() / decrement_async_work() should be used to ensure a graceful shutdown.
